Accessible Web Design

A guide to accessibility on the web, including the importance, challenges, and principles of accessible web design.

Introduction to Web Accessibility

Web accessibility involves making all digitial content accessible to a variety of users. As a tool for developers, the Web Content Accessibility Guidelines (WCAG) outlines principles that should be adhered to in order to make sure web pages are designed with accessibility built in.

Why is web accessibility important?

Content on the web should be universally accessible. Often times, however, people with disabilities struggle to access the same web content that abled users can. Web accessibility is important because it places a focus on making sure all users are given the same oppurtunity to engage with information on the web, regardless of their abilities.

What are some common challenges for users?

For blind or visually impaired users, challenges include not having access to visual content such as videos, images, text and other content that has been obscured. For deaf users, audio content such as videos and audio files may be inaccessible. In both cases it's important to have media alternatives.

What is WCAG?

WCAG stands for Web Content Accessibility Guidelines. They are guidelines that detail what is and isn't accessible in terms of web content.

History of WCAG

WCAG 1.0 was developed in 1999 as part of the Web Accessibility Initiative (WAI) by the World Wide Web Conortium (W3C). It provided clarity and structure for web developers by creating a rubric and testable criteria for success. Since then, WCAG has been updated several times. In 2008, WCAG 2.0 introduced the four POUR principles of web accessibility. It was expanded upon in 2018 with WCAG 2.1, the most modern version, which accounts for accessibility on mobile devices and other recent advances in technology that WCAG 2.0 could no longer keep up with.

Levels of Conformance

WCAG has three levels of conformance: A, AA, and AAA. Level A describes the minimum level of conformance. Level AA is generally adopted as the requirement for accessible web sites, and satisfies the criteria for both Level A and Level AA. Level AAA is the highest level of conformance, meeting the success criteria for all other levels. For some content, it is not possible to satisfy all Level AAA criteria.

Guidelines and Success Criteria

The guidelines and success criteria laid out in WCAG are meant to be tools for developers to test if their websites are accessible. In WCAG 2.1 there are three types of guidance provided: sufficient techniques, advisory techniques, and failures. While these are important considerations, it's worth noting that W3C reccomends prioritizing meeting the success criteria listed in WCAG over complying with the reccomended techniques. Other techniques can and should be used to make websites accessible.

What are POUR Principles?

POUR is an acronym that includes four principles for accessibility as outlined by WCAG. These four pillars encapsulate goals and criteria to ensure web pages are designed with accessibility in mind.

Everything on a webpage should be perceivable to the user, including the UI (User Interface) and page content. To acheive this, content must be perceivable through multiple means. This way, users who have one or more senses restricted are still aware of the content available.

    Example Problems
  • Audio content is not perceivable to deaf users
  • Images are not perceivable to blind users
  • Information conveyed through color is not perceivable to colorblind users
    Potential Solutions
  • Audio content should have a text alternative, such as transcrips and captioning on videos
  • Images should have alt text
  • Information should never rely soley on color to distinguish it

Users should be given multiple means to interact with a webpage. The UI should be accessible to everyone, whether they use a mouse, keyboard, mouth stick, or other tool to navigate a website. Under no circumstances should the UI require an action that a user is not able to perform.

    Example Problems
  • A button requires a click to perform an action
  • Content is timed, creating an obstacle for some users
  • Animation and flashing content prevents some users from being able to use the webpage
    Potential Solutions
  • Operable modalities should include clicks, keyboard entries, pointer gestures, etc
  • Preferable content would not be timed, but if so should allow for pauses, suppressed interruptions, and timeout warnings
  • Nothing should flash more than 3 times per second, and animation can be disabled

Website information, functions, and navigation should be intuitive for the user. Everything they need to know about the page, including content and operation, should be available from the page.

    Example Problems
  • The website uses tehnical jargon, slang, or abbreviations which are uninterpretable to the user
  • There is no perceivable change in content when focused on
  • Forms that require input do not specify what is needed from the user or offer validation/correction
    Potential Solutions
  • Websites should use language appropriate targetted language so that it can be undestand by the website's audience
  • Users should be able to understand where they are on the page, and content focused on should indicate location
  • Forms should identify errors and suggest solutions with clear lables and instructions for users

A website must be able to fit the current needs of the user, as well as anticipate and adapt to future needs. It should also be able to be reliable interpreted by the user through various means, including assistive technologies. Content should remain accessible and interpretable even as technology evolves.

    Example Problems
  • Website not compatible with older browsers
  • Website design not standard across platforms
  • Website not supported by multple assistive technologies
    Potential Solutions
  • HTML should be compatible across browser and OS versions
  • Website should be scalable and compatible with all platforms
  • Website should account for various user agents and be supported by all

Accessibility Testing Tools

Using testing tools is a great way to make sure that web pages are accessibile throughout the building and design process. Listed below are a few tools that may prove useful for checking web accessibility.

Siteimprove

Generates a Digital Certainty Index to see how web pages live up to standards, assessing Quality Assurance, Search Engine Optimization, and Accessibility.

Link to learn more about Siteimprove

Deque

Performs accessibility audits on contents of web pages and applications to ensure consistency with Web Content Accessibility Guidelines (WCAG).

Link to learn more about Deque

WAVE

Identifies accessibility and Web Content Accessibility Guidelines (WCAG) errors and facilitates human evaluation of web content.

Link to learn more about WAVE

TestLodge

Tests keyboard navigation accessibility to ensure that keyboard users can accesss the same features as those navigating web pages with a mouse.

Link to learn more about TestLodge

WebAIM

Checks the color contrast between items in the foreground and background to ensure legibility. Compares results to WCAG levels of conformance.

Link to learn more about WebAIM

NVDA

A free and portable screen reader that many blind users use to navigate websites. Available for PCs running Microsoft Windows 7 SP1 and later.

Link to learn more about NVDA
